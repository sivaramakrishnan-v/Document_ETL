# Technical Documentation: GenerationModelGateway

## 1. High-Level Overview
*   **What is the core business/architectural purpose of this specific service?**
    The `GenerationModelGateway` is a specialized gateway service that acts as the single, controlled entry point for all interactions with the primary Large Language Model (LLM) used for generating text. Its purpose is to abstract away the specifics of the LLM client, provide robust fault tolerance, and centralize token usage tracking. It ensures that if the external LLM API fails, the application can handle it gracefully without crashing.

*   **How does it fit into the RAG/ETL pipeline?**
    This service is the heart of the **generation** phase of the RAG pipeline. Any service that needs to generate text (e.g., `ChatService`, `AgenticRagService`) does not call the LLM directly. Instead, it calls this gateway.
    1.  **Input**: It receives a formatted prompt string.
    2.  **Process**: It sends the prompt to the underlying `ChatLanguageModel`.
    3.  **Output**: It returns the text response from the model and, crucially, records the number of input and output tokens used for cost and usage monitoring.

## 2. Core Dependencies & Frameworks
*   **`ChatLanguageModel` (from LangChain4j)**: This is the core dependency, an interface from the LangChain4j library that represents the connection to an external chat-based LLM (like GPT-4, Gemini, etc.). The specific implementation of this model is injected via Spring's `@Qualifier("generationChatLanguageModel")`.
*   **`TokenUsageManagerService`**: A critical dependency that this gateway calls after every successful or failed LLM interaction. This other service is responsible for counting the tokens in the prompt and response and logging them for cost analysis and rate-limiting purposes.
*   **`Resilience4j CircuitBreaker`**: All `generate` methods are wrapped in a circuit breaker named `"generationLlm"`. This is the most important feature of the gateway. If the external LLM API starts failing or timing out, the circuit breaker will "open," and subsequent calls will immediately fail and execute a `fallbackGenerate` method. This prevents a failing external API from cascading failures throughout the application.
*   **`@Value` Annotation**: Used to inject the configured model name (`app.rag.generation.model-name`) from the application properties. This is important for tagging token usage data with the correct model identifier.

## 3. Key Components & Internal Logic
*   **Inputs & Configuration:**
    *   `prompt` (String): The fully formatted text prompt to be sent to the LLM.
    *   `operationName` (String, optional): A descriptive name for the type of generation being performed (e.g., `"chat.answer"`, `"agent.rewrite_query"`), used for more granular token tracking.

*   **Core Workflows:**
    1.  **`generate(String prompt, ...)`**: This is the main public method.
        *   It directly calls the `generate` method on the injected `generationChatLanguageModel` bean.
        *   **On Success**: If the call succeeds, it immediately calls `tokenUsageManagerService.recordSuccess()`, passing the prompt, the response, and the model/operation name so the tokens can be counted and logged.
        *   **On Failure**: If the LLM call throws an exception, the Resilience4j circuit breaker intercepts it.
    2.  **`fallbackGenerate(...)`**: This is the circuit breaker's fallback method.
        *   It is only executed if the call to the LLM fails.
        *   It calls `tokenUsageManagerService.recordFailure()`, indicating that a request was attempted but failed. This is important for tracking error rates.
        *   It logs a warning that the model is unavailable.
        *   It throws a new, user-friendly `IllegalStateException` that wraps the original exception. This signals to the calling service that the generation failed and the operation should be aborted, but it does so in a controlled manner.

*   **Outputs/Side Effects:**
    *   **Return Value**: On success, returns the `String` response generated by the LLM. On failure, it throws an exception.
    *   **Side Effects**:
        *   Makes a network call to the external LLM API.
        *   Calls the `TokenUsageManagerService` to record the outcome (and token count) of the API call.

## 4. End-to-End Request Flow (How it runs)
1.  **Request In**: The `ChatService` has constructed a prompt and calls `generationModelGateway.generate(prompt, "chat.answer")`.
2.  **Processing (Success Case)**:
    a. The gateway calls the `generate` method on its `ChatLanguageModel` instance, which makes a network request to the configured LLM provider.
    b. The LLM provider returns a text answer.
    c. The gateway receives the answer and immediately calls `tokenUsageManagerService.recordSuccess("chat.answer", "gemini-1.5-pro", prompt, answer)`.
    d. The gateway returns the answer string to the `ChatService`.
3.  **Processing (Failure Case)**:
    a. The gateway calls the `generate` method, but the network request to the LLM provider times out.
    b. The circuit breaker intercepts the `TimeoutException`.
    c. The `fallbackGenerate` method is executed.
    d. It calls `tokenUsageManagerService.recordFailure(...)`.
    e. It throws a new `IllegalStateException`. The `ChatService` catches this exception and proceeds to its own error-handling logic (e.g., logging the failure to MLflow and returning an error to the user).
